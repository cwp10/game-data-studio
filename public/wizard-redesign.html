Project Wizard v3 — 2계층 설계 문서

2026-06-12 · Base Archetype 직접 선택 + Feature Overlay 조합 · Sonnet 시드 데이터 전용

1 · 핵심 문제 진단
기존 방식 (All-AI) — 구조 불안정
keywords[] → Haiku/Sonnet → 테이블+컬럼+타입+행+관계
✗ AI가 구조를 직접 결정 — 타입 오류, FK 불일치, 나쁜 설계 패턴
✗ 같은 키워드여도 매번 다른 테이블 구조 생성 (비결정론)
✗ "방치형" + "수집형" 동시 선택 시 성장 모델 충돌 불가피
✗ matcher.ts 점수 기반 추론 — 사용자 의도가 틀릴 수 있음
새 방식 (2계층 Code + AI) — 안정
Base 1개 선택 → genre-seeds.ts → 베이스 테이블+성장모델
기능 N개 선택 → modules.ts → 오버레이 테이블 병합
구조 확정 (즉시) + Sonnet → 시드 행 값만
✓ 성장 모델은 Base가 결정 — 방치형/수집형 충돌 원천 차단
✓ 오버레이는 베이스에 테이블 추가만 — 핵심 엔티티 건드리지 않음
✓ 구조는 즉시, 시드는 비동기 — UX 반응성 향상
★ 왜 matcher.ts가 필요 없는가 — 어드바이저 핵심 지적
성장 모델은 additive feature가 아닙니다. 수집형 RPG는 characters + character_levels(캡 있는 성장), 방치형 RPG는 heroes + dps_growth/cost_mult 컬럼 + 희소 체크포인트(무한 성장)입니다. 두 키워드를 동시에 선택하면 protagonist 테이블이 2개 생기거나 성장 모델이 충돌합니다 — mergeColumns()로 해결 불가.

따라서 성장 모델의 소유권은 Base Archetype에 있습니다. 사용자가 "베이스 장르를 직접 선택"하는 방식이 유일하게 안전합니다. matcher.ts 점수 추론 방식은 제거합니다.
2 · 2계층 아키텍처 — Base + Overlay
Layer 1 Base Archetype — 1개 선택 필수
수집형 RPG characters + character_levels (캡 성장, 지수 곡선)
방치형 RPG heroes(dps_growth·cost_mult) + 희소 체크포인트 (무한 성장)
MMORPG classes + class_levels + enhance_table
턴제/액션 RPG characters + character_levels + elements (속성상성)
로그라이크 RPG characters + floor_scaling (런 기반, 레이아웃 없음)
SRPG units(growth_hp·growth_atk·growth_def) + weapon_triangle
핵심 규칙: Base는 protagonist 엔티티와 성장 모델을 소유합니다.
이 선택이 바뀌면 테이블 구조가 근본적으로 달라집니다.
genre-seeds.ts가 이 층의 SSOT입니다.

FK 인텐트 처리 원칙
오버레이 모듈은 FK target을 후보 목록으로 선언합니다. 예: 가챠·뽑기의 gacha_pool_items.character_id → target_candidates: ["characters", "heroes", "units"]

assembleOverlays(baseGenre, features[]) 동작:
1. baseTables = SEED_TEMPLATES[baseGenre].map(t => t.name) — Set으로 변환
2. 각 feature의 테이블 순회: baseTables.has(table.name) → true면 skip (중복 방지)

4. fkIntents: target_candidates.find(t => allTableNames.has(t)) → 첫 매칭으로 relation 생성

7 · Sonnet 시드 프롬프트 전략
역할 분리 원칙 (v3에서도 동일)
코드가 테이블+컬럼+타입+FK를 확정한 후, Sonnet에게 "이미 확정된 구조에 값만 채워줘"를 요청합니다. 구조는 이미 알고 있으므로 Sonnet은 창의적 값 생성에만 집중 — 훨씬 단순하고 신뢰할 수 있는 프롬프트.

베이스 + 오버레이 조합에 따라 프롬프트가 동적으로 달라집니다: 방치형 + 프레스티지를 선택하면 "hero_levels는 희소 체크포인트만" + "prestige_bonuses는 레벨 1~10, gold_mult 2배 증가"가 프롬프트에 추가됩니다.
동적 프롬프트 조합 예시 (수집형 + 가챠·뽑기):

장르: 수집형 RPG / 추가 기능: 가챠·뽑기

아래 테이블들에 현실적인 게임 예시 데이터를 생성하라:
  characters: name(string), grade(string), element(string), base_hp(number), base_atk(number), ...
  character_levels: character_id(string), level(number), hp(number), atk(number), ...
  gacha_pools: pool_name(string), pool_type(string), pity_count(number), is_active(boolean)
  gacha_pool_items: pool_id(string), character_id(string), weight(number), guaranteed_pity(number)
  ...

★ 규칙:
- characters: id=char_001 형식, 5개 행 (SSR 1개, SR 2개, R 2개)
- character_levels: level=1,2,3,5,7,10,15,20,30,50 (10행, HP 레벨1=500~1500, 레벨50=15000~40000)
- gacha_pool_items: weight 합=100, character_id는 characters의 실제 id 값 사용
- 기타 테이블: 3-5개 행

출력: {"table_name":[{row},...], ...}
